運作原理
作業系統在開始運行使用者進程的時候，內核便開始處於被動狀態，只有在出現以下幾種情況的時候才會觸發硬體機制陷入內核：（1）用戶代碼由於某種原因引發異常（例如除以0）;（2）硬體產生中斷並且沒有遮罩觸發中斷；（3）使用者代碼調用相關指令（例如x86體系下的int系統調用指令）主動陷入內核。以上三種情況便是異常、中斷、系統調用機制。這三種機制由於需要陷入內核所以在進入內核之前必須先保存現場，然後回到用戶環境的時候恢復現場，xv6對著三種機制都採用相同的處理方式陷入內核

在x86體系下，這三種機制都會觸發相同的硬體操作，完成部分保護現場的任務，上圖中通過將部分暫存器值壓入堆疊來實現保護現場，如果觸發中斷（由於系統調用、中斷、異常具有相同的處理機制，所以一下全以中斷代稱）前處於內核態，則直接在當前state中保護現場，如果處於使用者態，則根據任務state描述符得到新的內核state並壓入用戶態的ss和esp。在硬體完成操作後，state中會得到以上資料。

x86規定了中斷、異常、系統調用的規範，在發生以上情況時，硬體能夠區分上述規定的256種觸發中斷的原因並通過尋找在記憶體中保存的中斷向量表來得到中斷處理常式的位址，然後將控制權交給中斷處理常式，這是x86架構下發生中斷時硬體完成的操作。

xv6通過一個存放函數指標的陣列來作為中斷向量表，在main函數初始化過程中，調用tvinit函數完成中斷向量表的初始化。tvinit將每一個中斷處理常式的位址寫入idt陣列中，idtinit將陣列位址載入中斷向量表暫存器，硬體能夠根據中斷向量表暫存器準確找出中斷處理常式，xv6使用vector.pl腳本生成vector陣列，vector陣列存放著每個中斷處理常式的入口位址，xv6簡單地將所有的中斷處理常式指向alltraps，由alltraps來負責具體的處理。在調用alltraps之前，xv6統一壓入errnum和trapnum來區分是256情況中的哪種。

在調用trap時由於trap是c函數所以會在state上壓入返回位元址eip和部分寄存值構成context結構，如下：

struct context {
uint edi;
uint esi;
uint ebx;
uint ebp;
uint eip;
};

之所以會重複壓入部分暫存器的值是應為Intel規定esi，ebx，esi是被調用者保存暫存器，需要由被調用者保存，而ebp則是c函數中維護每個函數state幀用的，eip是call指令壓state的，也就是說，當call指令執行後會執行以上動作才跳入trap函數體

暫存器使用慣例

在trap執行ret指令返回之前會彈出context恢復state的調用前狀態。

明確context結構能夠使我們很容易理解進程調用時，通過切換上下文context，很容易便能實現切換進程的目的，因為任何C函數的調用都會形成context結構，在進程調度時通過切換context結構，很容易使得函數返回時返回到另一個進程中去

實際調度的情況是這樣的，當把當前進程上下文切換至另一個context上下文時，調度器會手動彈出context結構來類比函數返回的情形，當彈出eip的時候就意味著回到了就緒進程的下一條指令中來。這樣便能實現進程的切換。

第一次運行進程

通過自建一個“現場”來類比返回第一個進程，手動寫入了trapframe和context結構的上下文來讓調度器調度並返回使用者進程，這些操作由allocproc負責，allocproc手動構建內核state並寫入內核暫存器來構成進程第一次運行的“現場”，如下面代碼所示：

// Allocate kernel stack.
if((p->kstack = kalloc()) == 0){
p->state = UNUSED;
return 0;
}
sp = p->kstack + KSTACKSIZE;

// Leave room for trap frame.
sp -= sizeof p->tf;
p->tf = (struct trapframe)sp;

// Set up new context to start executing at forkret,
// which returns to trapret.
sp -= 4;
(uint)sp = (uint)trapret;

sp -= sizeof p->context;
p->context = (struct context)sp;
memset(p->context, 0, sizeof *p->context);
p->context->eip = (uint)forkret;

allocproc在“正常現場”中本應該壓入esp的地方壓入trapret，trapret處的代碼負責彈出暫存器恢復現場，然後在本應該是trap返回位址的地方壓入forkret的位址並將context的內容置零，通過這種方式，調度器在調度的時候手動彈出context，但是此時彈出的eip並不是trap的返回地址而是forkret，forkret返回時彈出壓入的trapret恢復現場，這樣進程第一次便開始運行了。

第一個使用者進程

在 main 初始化了一些設備和子系統後，它通過調用 userinit建立了第一個進程。userinit首先調用上面講的allocproc來創建進程的內核state，來完成進程運行的必要環境。userinit 僅僅在創建第一個進程時被調用，而 allocproc 創建每個進程時都會被調用。然後userinit設置全域靜態變數initproc的值，設置頁表中的內核態映射，然後調用inituvm將initcode.S移動到虛擬位址0處，並設置頁表，然後為trapframe賦 初始值，最後使p->state設置為RUNNABLE狀態，以便調度器能夠調度運行。

initcode.S剛開始會將 argv，init，0 三個值推入state中，接下來把 %eax 設置為 SYS_exec 然後執行 int T_SYSCALL：這樣做是告訴內核運行 exec 這個系統調用。如果運行正常的話，exec 不會返回：它會運行名為 init 的程式，init 是一個以空字元結尾的字串，即 /init。如果 exec 失敗並且返回了，initcode 會不斷調用一個不會返回的系統調用 exit 。

系統調用 exec 的參數是 init、argv。最後的0讓這個手動構建的系統調用看起來就像普通的系統調用一樣。和之前的代碼一樣，避免為第一個進程的運行單獨寫一段代碼，而是儘量使用通用于普通操作的代碼。

現在 initcode 已經執行完了，進程將要運行 /init。 init會在需要的情況下創建一個新的控制台設備檔，然後把它作為描述符0，1，2打開。接下來它將不斷迴圈，開啟控制台 shell，處理沒有父進程的僵屍進程，直到 shell 退出，然後再反復。系統就這樣建立起來了。

C中斷處理常式trap

trap函數主要根據trapframe中的trapno來確定到底是哪種原因導致中斷的發生，如果是系統調用，則通過調用syscall函數負責具體的系統調用處理

if(tf->trapno == T_SYSCALL){
if(proc->killed)
exit();
proc->tf = tf;
syscall();
if(proc->killed)
exit();
return;
}

在這裡處理proc->killed的原因是kill系統調用的需要，kill系統調用通過將killed置1，來殺死一個進程，由於遲早進程會由於系統調用或者時鐘中斷進入trap，此時trap檢查到killed置1便能夠將進程殺死。

這裡重新更新了tf的位址，這樣做的原因是trapframe的大小由於可能硬體未壓入ss和esp導致不一致，而任務statets總是指向內核state所在頁的最高地址處：

cpu->ts.esp0 = (uint)proc->kstack + KSTACKSIZE;
proc->tf最開始的賦值是按照大小包括esp和ss來的，在allocproc中：
p->tf = (struct trapframe*)sp;

所以在這裡重新設置了tf的位址。

如果中斷產生的原因是硬體中斷或者異常，trap則調用相應的函數來進行處理：

系統調用實現機制

syscall通過trapframe中的eax來確定系統調用號以決定調用那個系統函數，當然eax的值或許是庫函數在調用int指令的時候設置的，只是保護現場使得存放在了trapframe中，然後通過系統調用號調用具體的系統調用處理函數並返回到trapframe中的eax位置，這樣恢復現場時庫函數便能根據eax得到系統調用的返回值

oid
syscall(void)
{
int num;

num = proc->tf->eax;
if(num > 0 && num < NELEM(syscalls) && syscalls[num]) {
proc->tf->eax = syscallsnum;
} else {
cprintf("%d %s: unknown sys call %d\n",
proc->pid, proc->name, num);
proc->tf->eax = -1;
}
}

參數或許是庫函數在陷入內核前壓state的，所以可以根據trapframe中的使用者stateesp來找到各種參數，使用了工具函數 argint、argptr 和 argstr來獲得第 n 個系統調用參數

他們分別用於獲取整數，指標和字串起始位址。argint 利用用戶空間的 %esp 寄存器定位第 n 個參數：%esp 指向系統調用結束後的返回位址。參數就恰好在 %esp 之上（%esp+4）。因此第 n 個參數就在 %esp+4+4*n。

argint 調用 fetchint 從使用者記憶體位址讀取值到 *ip。fetchint 可以簡單地將這個位址直接轉換成一個指標，因為用戶和內核共用同一個頁表，但是內核必須檢驗這個指標的確指向的是使用者記憶體空間的一部分。內核已經設置好了頁表來保證本進程無法訪問它的私有位址以外的記憶體：如果一個用戶嘗試讀或者寫高於（包含）p->sz的位址，處理器會產生一個段中斷，這個中斷會殺死此進程，正如我們之前所見。但是現在，我們在內核態中執行，用戶提供的任何位址都是有權訪問的，因此必須要檢查這個位址是在 p->sz 之下的。

argptr 和 argint 的目標是相似的：它解析第 n 個系統調用參數。argptr 調用 argint 來把第 n 個參數當做是整數來獲取，然後把這個整數看做指標，檢查它的確指向的是用戶位址空間。argptr 的源碼中有兩次檢查。首先，用戶的state指標在獲取參數的時候被檢查。然後這個獲取到得參數作為用戶指標又經過了一次檢查。

argstr 是最後一個用於獲取系統調用參數的函數。它將第 n 個系統調用參數解析為指標。它確保這個指標是一個 NUL 結尾的字串並且整個完整的字串都在使用者位址空間中。

系統調用的實現（例如，sysproc.c 和 sysfile.c）僅僅是封裝而已：他們用 argint，argptr 和 argstr 來解析參數，然後調用真正的實現。

// User code makes a system call with INT T_SYSCALL.
// System call number in %eax.
// Arguments on the stack, from the user call to the C
// library system call function. The saved user %esp points
// to a saved program counter, and then the first argument.

// Fetch the int at addr from the current process.
int
fetchint(uint addr, int *ip)
{
if(addr >= proc->sz || addr+4 > proc->sz)
return -1;
*ip = (int)(addr);
return 0;
}

// Fetch the nul-terminated string at addr from the current process.
// Doesn’t actually copy the string - just sets *pp to point at it.
// Returns length of string, not including nul.
int
fetchstr(uint addr, char **pp)
{
char *s, *ep;

if(addr >= proc->sz)
return -1;
pp = (char)addr;
ep = (char*)proc->sz;
for(s = *pp; s < ep; s++)
if(*s == 0)
return s - *pp;
return -1;
}

// Fetch the nth 32-bit system call argument.
int
argint(int n, int ip)
{
return fetchint(proc->tf->esp + 4 + 4n, ip);
}

// Fetch the nth word-sized system call argument as a pointer
// to a block of memory of size bytes. Check that the pointer
// lies within the process address space.
int
argptr(int n, char **pp, int size)
{
int i;

if(argint(n, &i) < 0)
return -1;
if(size < 0 || (uint)i >= proc->sz || (uint)i+size > proc->sz)
return -1;
pp = (char)i;
return 0;
}

// Fetch the nth word-sized system call argument as a string pointer.
// Check that the pointer is valid and the string is nul-terminated.
// (There is no shared writable memory, so the string can’t change
// between this check and being used by the kernel.)
int
argstr(int n, char **pp)
{
int addr;
if(argint(n, &addr) < 0)
return -1;
return fetchstr(addr, pp);
}
