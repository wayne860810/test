1. Locking
    xv6支持多核心CPU,計算機上有多個同時運行代碼的CPU,但是所有CPU共享同一個地址空間,為了保護數據結構的一致性,xv6需要某一種機制來防止它們互相干擾。其實即使在單處理器中,也同樣需要處理內核數據結構的互斥現象。
    xv6以硬體提供的處理方式,採用了較為基本的鎖機制來處理多核操作系統的互斥問題,xv6只是簡單的使用了spinlock ,因為一般情況下內核對數據結構的佔用時間一般比較短,使用spinlock 能夠簡單的實現和減少處理同步時容易出現的deadlock問題。

  2. Spinlock

// Mutual exclusion lock.
struct spinlock {
  uint locked;       // Is the lock held?

  // For debugging:
  char *name;        // Name of lock.
  struct cpu *cpu;   // The cpu holding the lock.
  uint pcs[10];      // The call stack (an array of program counters)
                     // that locked the lock.
};

其中locked 代表鎖的狀態若為false 表示鎖尚未鎖住並能夠讀取地址數據,除此之外spinlock 中還有附加其他訊息,例如:name代表鎖的名字、cpu代表擁有鎖的cpu和pcs呼叫堆疊。
即使如此,在程式中還是有可能發生有兩個CPU同時讀取locked的值為false,並認為都可以獲得鎖佔用數據結構,然後將locked 佔用,實際上有兩個CPU同時獲得了鎖,出現這種現象的主因是因為讀取和修改不是一個atomic operation,如果讀取和修改這兩個過程連續進行不可打斷,並在訪問時只允許一個CPU進行,便能實現訪問locked的atomic operation,xv6使用了xchg指令實現,xchg交換一個寄存器和內存字的值,通過循環反复xchg,如果返回的內存字值為true ,代表有CPU或者進程佔用鎖,繼續循環等待不斷xchg,如果xchg返回為false ,代表目前沒有人佔用鎖,通過將鎖置1佔用數據結構,然後跳出循環,如此一來便能在讀取某個數據地址時提前獲得鎖,若無法拿到的話就不停地等待,直到鎖的使用權到手。

//xchg//
static inline uint xchg(volatile uint *addr, uint newval)
{
uint result;

// The + in "+m" denotes a read−modify−write operand.
asm volatile("lock; xchgl %0, %1" :
"+m" (*addr), "=a" (result) :
"1" (newval) :
"cc");
return result;
}
  1. Acquire和release
    xv6使用acquire和release操作來代表獲得鎖和解除鎖的佔用。
//acquire//
void acquire(struct spinlock *lk)
{
   pushcli(); // 禁用中斷以避免死鎖。
  if(holding(lk))
    panic("acquire");

  // The xchg is atomic.
  while(xchg(&lk->locked, 1) != 0);//等待鎖的使用權

  
  __sync_synchronize();//確保在獲得鎖前程式都完成

  //紀錄鎖的資訊好偵錯
  lk->cpu = cpu;
  getcallerpcs(&lk, lk->pcs);
  }

//release//
void release(struct spinlock *lk)
 {
 if(!holding(lk))
 panic("release");

 lk−>pcs[0] = 0;
 lk−>cpu = 0;
 
 __sync_synchronize(); //在完成指令前不讓其他程式讀取


 asm volatile("movl $0, %0" : "+m" (lk−>locked) : );

 popcli();
 }
  1. Pushcli和 popcli
    xv6在acquire中只是簡單地關閉所有的中斷,之所以需要關閉中斷是為了防止當在搶占式kernel中,當前過程佔用鎖然後進入中斷處理再次想要獲得鎖時出現的deadlock現象,使用spinlock非常容易出現deadlock現象,尤其是在搶占式kernel中,所以xv6在中斷處理和spinlock的關係是佔用鎖時禁止所有中斷。
    pushcli主要關閉外部中斷並遞增使用pushcli關閉中斷的次數,這樣做的原因是如果指令獲得了兩個鎖,那麼只有當兩個鎖都被釋放後中斷才會被允許。同時acquire 一定要在可能獲得鎖的 xchg 之前使用 pushcli,保證中斷被禁用。如果兩者顛倒了,就可能在幾個時鐘週期裡,中斷仍被允許,而鎖也被獲得了,如果此時不幸地發生了中斷,系統就會死鎖。類似的,release 也一定要在釋放鎖的 xchg 之後調用 popcli。
void pushcli(void)
{
  int eflags;

  eflags = readeflags();
  cli();
  if(cpu->ncli == 0)
    cpu->intena = eflags & FL_IF;
  cpu->ncli += 1;
}

void popcli(void)
{
  if(readeflags()&FL_IF)
    panic("popcli - interruptible");
  if(--cpu->ncli < 0)
    panic("popcli");
  if(cpu->ncli == 0 && cpu->intena)
    sti();
}

  2. 結論
    xv6 非常謹慎地使用鎖來避免競爭條件。使用鎖的一個難點在於要決定使用多少個鎖,以及每個鎖保護哪些數據、不變量。不過有幾個基本原則。
    (1)當一個CPU正在寫一個變量,而同時另一個CPU可能讀/寫該變量時,需要用鎖防止兩個操作重疊。
    (2)當用鎖保護不變量時,如果不變量涉及到多個數據結構,通常每個數據結構都需要用一個單獨的鎖保護起來,這樣才能維持不變量。

    如果一段程式要使用多個鎖,那麼必須要注意程式每次運行都要以相同的順序獲得鎖,否則就有死鎖的危險。假設某段程式的兩條執行路徑都需要鎖A和鎖B但路徑1獲得鎖的順序是 A->B,而路徑2獲得鎖的順序是B->A。這樣就有能路徑1獲得了鎖A,而在它繼續獲得鎖B之前,路徑2獲得了鎖B,這樣就死鎖了。這時兩個路徑都無法繼續執行下去了,因為這時路徑1需要鎖B,但鎖B已經在路徑2手中了,反之路徑2也得不到鎖A。為了避免這種死鎖,所有的程式路徑獲得鎖的順序必須相同。避免死鎖也是我們把鎖作為函數使用規範的一部分的原因：調用者必須以固定順序調用函數,這樣函數才能以相同順序獲得鎖。
