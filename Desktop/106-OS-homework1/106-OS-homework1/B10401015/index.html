Ch.2 Page tables

xv6通過頁表機制實現了對內存空間的控制。頁表使得 xv6 能夠讓不同進程各自的地址空間映射到相同的物理內存上，還能夠為不同進程的內存提供保護。 除此之外，我們還能夠通過使用頁表來間接地實現一些特殊功能。xv6 主要利用頁表來區分多個地址空間，保護內存。另外，它也使用了一些簡單的技巧，即把不同地址空間的多段內存映射到同一段物理內存（內核部分），在同一地址空間中多次映射同一段物理內存（用戶部分的每一頁都會映射到內核部分），以及通過一個沒有映射的頁保護用戶棧。
在xv6中，除了每個CPU獨立的數據具有非零的地址外，其餘包括內核數據段，內核代碼段，用戶數據段，用戶代碼段都是地址為0的描述，這樣大大簡化了地址轉換操作和程序的編程，因為分段機制是由程序員控制，而分頁機制是由作業系統負責的，在xv6中，可以簡化為以下地址映射：

虛擬地址——分—頁——>物理地址

分頁機制是通過頁表來進行轉換的，具體轉換關係如下圖：

x86中所有的虛擬地址都經過頁表來完成地址轉換，頁表由cr3寄存器指定的物理地址來表示，內存地址轉換單元mmu通過查找頁表來確定最後的物理地址，通過給cr3賦值便能實現不同進程擁有不同的頁表，也就是不同進程擁有不同的地址空間。
頁表由一級的頁目錄項和二級的頁表項組成，每個頁目錄項下級有1024個連續的頁表項（每個頁表項4Byte，剛好佔用4K空間，也就是一頁），頁目錄項同時也是連續的，一共有1024個頁目錄項，由於32位系統地址線只有32位，所以最高支持4G的地址空間，頁目錄項也是連續的，所以頁目錄剛好也佔用一頁。
每個頁目錄項和頁表項由下一級的物理地址和相關標誌位組成，頁目錄項擁有下一級頁表的物理地址，頁表項擁有實際物理地址的部分，通過設置權限位可以實現內核和用戶進程代碼和數據的保護。
虛擬地址前十位作為頁目錄的偏移找到頁目錄項，找到頁表基地址，接下來十位作為頁表項的偏移找到頁表項，最後將頁表項的基址+最後12位偏移得到實際的物理地址。

xv6內存初始化
在初始化main函數最開始處，xv6具有以下的內存佈局：
內核代碼存在於物理地址低地址的0x100000處，頁表為main.c文件中的entrypgdir，其中虛擬地址低4M映射物理地址低4M，虛擬地址 [KERNBASE, KERNBASE+4MB) 映射到 物理地址[0, 4MB)。可見現在內存實際能用的虛擬地址空間顯然是不足以完成正常工作的，所以初始化過程中需要重新設置頁表。

1.物理內存的初始化
xv6在main函數中調用kinit1和kinit2來初始化物理內存，kinit1初始化內核末尾到物理內存4M的物理內存空間為未使用
kinit1(end, P2V(410241024)); // phys page allocator
kinit2初始化剩餘內核空間到PHYSTOP為未使用
kinit2(P2V(410241024), P2V(PHYSTOP)); // must come after startothers()
兩者的區別在於kinit1調用前使用的還是最初的頁表（也就是是上面的內存佈局），所以只能初始化4M，同時由於後期再構建新頁表時也要使用頁表轉換機制來找到實際存放頁表的物理內存空間，這就構成了自舉問題，xv6通過在main函數最開始處釋放內核末尾到4Mb的空間來分配頁表，由於在最開始時多核CPU還未啓動，所以沒有設置鎖機制。
initlock(&kmem.lock, “kmem”);
kmem.use_lock = 0;
freerange(vstart, vend);
kinit2在內核構建了新頁表後，能夠完全訪問內核的虛擬地址空間，所以在這裡初始化所有物理內存，並開始了鎖機制保護空閒內存鏈表。
void
kinit2(void *vstart, void *vend)
{
freerange(vstart, vend);
kmem.use_lock = 1;
}

物理內存管理
xv6對上層提供kalloc和kfree接口來管理物理內存，上層無需知道具體的細節，kalloc返回虛擬地址空間的地址，kfree以虛擬地址為參數，通過kalloc和kfree能夠有效管理物理內存，讓上層只需要考慮虛擬地址空間。
xv6通過將未分配的內存構成一個簡單的鏈表來管理物理內存，具體的資料結構如下：
struct run {
struct run *next;
};
struct {
struct spinlock lock;
int use_lock;
struct run freelist;
} kmem;
xv6使用了空閒內存的前部分作為指針域來指向下一頁空閒內存，物理內存管理是以頁（4K）為單位進行分配的。也就是說物理內存空間上空閒的每一頁，都有一個指針域（虛擬地址）指向下一個空閒頁，最後一個空閒頁為NULL
通過這種方式，只需要保存著虛擬地址空間上的freelist地址即可，kalloc和kfree操作的地址都是虛擬地址，那麼為什麼就能夠準確找到每一物理頁的位置呢，這是還是通過頁表來完成的，仔細看內核映射佈局
{ (void)data, V2P(data), PHYSTOP, PTE_W}, // kern data+memory
由此可知 VA’s [KERNBASE, KERNBASE+PHYSTOP) to PA’s [0, PHYSTOP)，
所以可以想成就算kalloc和kfree操作的是虛擬地址，但是也能夠總是準確找到每一頁的位置，其實也不用關心kalloc和kfree是如何組織空閒頁的，只要保證正確並能供上層調用即可，上層無需關心是如何實現的。上層調用的只需要想著「虛擬地址a對應的一頁釋放為空閒頁」「分配一頁返回虛擬地址給我」即可

xv6內存管理函數
xv6通過提供幾個接口來實現內核頁表的控制和用戶頁表的控制，xv6讓每個進程都有獨立的頁表結構，在切換進程時總是需要切換頁表，切換頁表的接口如下：
// Switch h/w page table register to the kernel-only page table,
// for when no process is running.
void
switchkvm(void)
{
lcr3(V2P(kpgdir)); // switch to the kernel page table
}
// Switch TSS and h/w page table to correspond to process p.
void
switchuvm(struct proc p)
{
pushcli();
cpu->gdt[SEG_TSS] = SEG16(STS_T32A, &cpu->ts, sizeof(cpu->ts)-1, 0);
cpu->gdt[SEG_TSS].s = 0;
cpu->ts.ss0 = SEG_KDATA << 3;
cpu->ts.esp0 = (uint)proc->kstack + KSTACKSIZE;
// setting IOPL=0 in eflags and iomb beyond the tss segment limit
// forbids I/O instructions (e.g., inb and outb) from user space
cpu->ts.iomb = (ushort) 0xFFFF;
ltr(SEG_TSS << 3);
if(p->pgdir == 0)
panic(“switchuvm: no pgdir”);
lcr3(V2P(p->pgdir)); // switch to process’s address space
popcli();
}
switchkvm簡單地將kpgdir設置為cr3寄存器的值，這個頁表僅僅在 scheduler內核線程中使用。
頁表和內核棧都是每個進程獨有的，xv6使用資料結構proc將它們統一起來，在進程切換的時候，他們也往往隨著進程切換而切換，內核中模擬出了一個內核線程，它獨佔內核棧和內核頁表kpgdir，它是所有進程調度的基礎。
switchuvm通過傳入的proc結構負責切換相關的進程獨有的資料結構，其中包括TSS相關的操作，然後將進程特有的頁表載入cr3寄存器，完成設置進程相關的虛擬地址空間環境。
進程的頁表在使用前往往需要初始化，其中必須包含內核代碼的映射，這樣進程在進入內核時便不需要再次切換頁表，進程使用虛擬地址空間的低地址部分，高地址部分留給內核，設置頁表時通過調用setupkvm、allocuvm、deallocuvm接口完成相關操作
setupkvm通過kalloc分配一頁內存作為頁目錄，然後將按照kmap資料結構映射內核虛擬地址空間到物理地址空間。期間調用了工具函數mappages。
// Set up kernel part of a page table.
pde_t
setupkvm(void)
{
pde_t pgdir;
struct kmap k;
if((pgdir = (pde_t)kalloc()) == 0)
return 0;
memset(pgdir, 0, PGSIZE);
if (P2V(PHYSTOP) > (void)DEVSPACE)
panic(“PHYSTOP too high”);
for(k = kmap; k < &kmap[NELEM(kmap)]; k++)
if(mappages(pgdir, k->virt, k->phys_end - k->phys_start,
(uint)k->phys_start, k->perm) < 0)
return 0;
return pgdir;
}
allocuvm、deallocuvm負責完成用戶進程的內存空間，allocuvm在設置頁表的同時還會分配物理內存供用戶進程使用allocuvm中的參數oldsz，newsz，意思就是分配虛擬地址oldsz到newsz的以頁為單位的內存，deallocuvm則相反，它將newsz到oldsz對應的虛擬地址空間內存置為空閒。
int
allocuvm(pde_t *pgdir, uint oldsz, uint newsz)
{
char mem;
uint a;
if(newsz >= KERNBASE)
return 0;
if(newsz < oldsz)
return oldsz;
a = PGROUNDUP(oldsz);
for(; a < newsz; a += PGSIZE){
mem = kalloc();
if(mem == 0){
cprintf(“allocuvm out of memory\n”);
deallocuvm(pgdir, newsz, oldsz);
return 0;
}
memset(mem, 0, PGSIZE);
if(mappages(pgdir, (char)a, PGSIZE, V2P(mem), PTE_W|PTE_U) < 0){
cprintf(“allocuvm out of memory (2)\n”);
deallocuvm(pgdir, newsz, oldsz);
kfree(mem);
return 0;
}
}
return newsz;
}
int
deallocuvm(pde_t *pgdir, uint oldsz, uint newsz)
{
pte_t pte;
uint a, pa;
if(newsz >= oldsz)
return oldsz;
a = PGROUNDUP(newsz);
for(; a < oldsz; a += PGSIZE){
pte = walkpgdir(pgdir, (char)a, 0);
if(!pte)
a += (NPTENTRIES - 1) * PGSIZE;
else if((*pte & PTE_P) != 0){
pa = PTE_ADDR(*pte);
if(pa == 0)
panic(“kfree”);
char *v = P2V(pa);
kfree(v);
*pte = 0;
}
}
return newsz;
}
到了這裡，xv6內存管理提供的主要接口也差不多了，兩個切換接口，一個設置內核頁表的接口，兩個為用戶進程管理內存，設置用戶代碼頁表的接口，xv6 vm.c文件中還提供了loaduvm將文件系統上的i節點內容讀取載入到相應的地址上，通過allocuvm接口為用戶進程分配內存和設置頁表，然後調用loaduvm接口將文件系統上的程序載入到內存，便能夠為exec系統調用提供接口，為用戶進程的正式運行做準備。
// Load a program segment into pgdir. addr must be page-aligned
// and the pages from addr to addr+sz must already be mapped.
int
loaduvm(pde_t *pgdir, char *addr, struct inode *ip, uint offset, uint sz)
{
uint i, pa, n;
pte_t *pte;
if((uint) addr % PGSIZE != 0)
panic(“loaduvm: addr must be page aligned”);
for(i = 0; i < sz; i += PGSIZE){
if((pte = walkpgdir(pgdir, addr+i, 0)) == 0)
panic(“loaduvm: address should exist”);
pa = PTE_ADDR(*pte);
if(sz - i < PGSIZE)
n = sz - i;
else
n = PGSIZE;
if(readi(ip, P2V(pa), offset+i, n) != n)
return -1;
}
return 0;
}
vm.C中還有一個inituvm函數，為第一個進程所使用，通過調用它能夠初始化虛擬地址為0的initcode.S的虛擬地址環境，initcode.S是獨立於內核編譯和鏈接的，它的加載地址和運行地址都為0。
// Load the initcode into address 0 of pgdir.
// sz must be less than a page.
void
inituvm(pde_t *pgdir, char *init, uint sz)
{
char mem;
if(sz >= PGSIZE)
panic(“inituvm: more than a page”);
mem = kalloc();
memset(mem, 0, PGSIZE);
mappages(pgdir, 0, PGSIZE, V2P(mem), PTE_W|PTE_U);
memmove(mem, init, sz);
}
當進程銷毀需要回收內存時，可以調用freevm清除用戶進程相關的內存環境，freevm首先調用deallocuvm將0到KERNBASE的虛擬地址空間回收，然後銷毀整個進程的頁表
// Free a page table and all the physical memory pages
// in the user part.
void
freevm(pde_t pgdir)
{
uint i;
if(pgdir == 0)
panic(“freevm: no pgdir”);
deallocuvm(pgdir, KERNBASE, 0);
for(i = 0; i < NPDENTRIES; i++){
if(pgdir[i] & PTE_P){
char * v = P2V(PTE_ADDR(pgdir[i]));
kfree(v);
}
}
kfree((char)pgdir);
}
在vm.c中，copyuvm負責複製一個新的頁表並分配新的內存，新的內存佈局和舊的完全一樣，xv6使用這個函數作為fork的底層實現
// Given a parent process’s page table, create a copy
// of it for a child.
pde_t
copyuvm(pde_t *pgdir, uint sz)
{
pde_t *d;
pte_t *pte;
uint pa, i, flags;
char *mem;
if((d = setupkvm()) == 0)
return 0;
for(i = 0; i < sz; i += PGSIZE){
if((pte = walkpgdir(pgdir, (void *) i, 0)) == 0)
panic(“copyuvm: pte should exist”);
if(!(pte & PTE_P))
panic(“copyuvm: page not present”);
pa = PTE_ADDR(pte);
flags = PTE_FLAGS(pte);
if((mem = kalloc()) == 0)
goto bad;

}
在vm.c的最後，有兩個函數，其中uva2ka將一個用戶地址轉化為內核地址，也就是通過用戶地址找到對應的物理地址，然後退出這個物理地址在內核頁表中的虛擬地址並返回，copyout則調用uva2ka則拷貝p地址len字節到用戶地址va中。
// Map user virtual address to kernel address.
char
uva2ka(pde_t *pgdir, char *uva)
{
pte_t *pte;
pte = walkpgdir(pgdir, uva, 0);
if((*pte & PTE_P) == 0)
return 0;
if((pte & PTE_U) == 0)
return 0;
return (char)P2V(PTE_ADDR(*pte));
}
// Copy len bytes from p to user address va in page table pgdir.
// Most useful when pgdir is not the current page table.
// uva2ka ensures this only works for PTE_U pages.
int
copyout(pde_t *pgdir, uint va, void *p, uint len)
{
char buf, pa0;
uint n, va0;
buf = (char)p;
while(len > 0){
va0 = (uint)PGROUNDDOWN(va);
pa0 = uva2ka(pgdir, (char)va0);
if(pa0 == 0)
return -1;
n = PGSIZE - (va - va0);
if(n > len)
n = len;
memmove(pa0 + (va - va0), buf, n);
len -= n;
buf += n;
va = va0 + PGSIZE;
}
return 0;
}

結尾
xv6對於物理內存的管理較為簡單，只是將每一空閒頁用鏈錶鏈接起來，向上提供kalloc和kfree接口來屏蔽管理物理內存的細節，xv6將內存管理分為內核地址空間管理和用戶地址空間管理，並提供幾個函數供系統調用過程調用，很多需要管理內存的系統函數例如exec，fork都需要使用到這些接口，vm.c和kalloc.c包含了內存管理的大部分內容，系統調用過程使用這些函數來初始化和處理頁表結構。
