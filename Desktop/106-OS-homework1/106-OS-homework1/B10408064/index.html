Ch5 Scheduling

運作原理

任一作業系統都可能執行超過電腦所擁有的處理器數目的進程(process)，因此處理器需要有一個給予進程適當時間的計畫。理想的分享機制是對進程透明。通常我們對進程造成一個自己獨佔處理器的假像，然後讓作業系統的多工機制（multiplex）將單獨的一個物理處理器類比為多個虛擬處理器。

Multiplex

Xv6 中多工的實現如下：當一個進程等待磁片請求時，Xv6 使之進入睡眠狀態，然後調度執行另一個進程。另外，當一個進程耗盡了它在處理器上運行的時間片（100毫秒）後，Xv6 使用時鐘中斷強制它停止運行，這樣調度器才能調度運行其他進程。這樣的多工機制為進程提供了獨佔處理器的假像，類似於 Xv6 使用記憶體分配器和分頁表為進程提供了獨佔記憶體的假像。

實現多工有幾個難點。首先，應該如何從運行中的一個進程切換到另一個進程？Xv6 採用了普通的上下文切換機制；雖然這裡的思想是非常簡潔明瞭的，但是其代碼實現是作業系統中最晦澀難懂的一部分。第二，如何讓上下文切換透明化？Xv6 只是簡單地使用時鐘中斷處理常式來驅動上下文切換。第三，可能出現多個 CPU 同時切換進程的情況，那麼我們必須使用一個帶鎖的方案來避免競爭。第四，進程退出時必須釋放其佔用記憶體與資源，但由於它本身在使用自己的資源（譬如其kernel stack），所以不能由該進程本身釋放其佔有的所有資源。Xv6 希望能夠簡潔明瞭地處理這些難點，不過最後其代碼實現還是比較巧妙。

Xv6 必須為進程提供互相協作的方法。譬如，父進程需要等待子進程結束，以及讀取管道資料的進程需要等待其他進程向管道中寫入資料。與其讓這些等待中的進程消耗 CPU 資源，不如讓它們暫時放棄 CPU，進入睡眠狀態來等待其他進程發出事件來喚醒它們。但我們必須要小心設計以防睡眠進程遺漏事件通知。

代碼解釋

Code: Context switching

如圖表5-1所示，xv6 在低層次中實現了兩種上下文切換：從進程的內核執行緒切換到當前 CPU 的調度器執行緒，從調度器執行緒到進程的內核執行緒。xv6 永遠不會直接從使用者態進程切換到另一個使用者態進程；這種切換是通過用戶態-內核態切換（系統調用或中斷）、切換到調度器、切換到新進程的內核執行緒、最後這個陷入返回實現的。

每個 xv6 進程都有自己的內核堆疊以及寄存器集合。每個 CPU 都有一個單獨的調度器執行緒，這樣調度就不會發生在進程的內核執行緒中，而是在此調度器執行緒中。執行緒的切換涉及到了保存舊執行緒的 CPU 寄存器，恢復新執行緒之前保存的寄存器；其中 %esp 和 %eip 的變換意味著 CPU 會切換運行堆疊與運行代碼。
swtch 並不瞭解執行緒，它只是簡單地保存和恢復寄存器集合，即上下文。當進程讓出 CPU 時，進程的內核執行緒調用 swtch 來保存自己的上下文然後返回到調度器的上下文中。每個上下文都是以結構體 struct context* 表示的，這實際上是一個保存在內核堆疊中的指標。swtch 有兩個參數：struct context **old、struct context *new。它將當前 CPU 的寄存器壓入堆疊中並將堆疊指針保存在 old 中。然後 swtch 將 new 拷貝到 %esp中，彈出之前保存的寄存器，然後返回。
回到使用者進程中，在中斷的最後，trap 會調用 yield。yield 又調用 sched，其中 sched 會調用 swtch 來保存當前上下文到 proc->context 中然後切換到之前保存的調度器上下文 cpu->scheduler。
swtch一開始從堆疊中彈出參數，放入寄存器 %eax 和 %edx中；swtch 必須在改變堆疊指標以及無法獲得 %esp 前完成這些事情。然後 swtch 壓入寄存器，在當前堆疊上建立一個新的上下文結構。僅有被調用者保存的寄存器此時需要被保存；按照 x86 的慣例即 %ebp %ebx %esi %ebp %esp。swtch 顯式地壓入前四個寄存器；最後一個則是在 struct context 被寫入 old時隱式地保存的，還有一個重要的寄存器，即程式計數器 %eip，該寄存器在使用 call 調用 swtch 時就保存在堆疊中 %ebp 之上的位置上了。保存了舊寄存器後，swtch 就準備要恢復新的寄存器了。它將指向新上下文的指標放入堆疊指標中。新的堆疊結構和舊的堆疊相同，因為新的上下文其實是之前某次的切換中的舊上下文。所以 swtch 就能顛倒一下保存舊上下文的順序來恢復新上下文。它彈出 %edi %esi %ebx %ebp 然後返回。由於 swtch 改變了堆疊指標，所以這時恢復的寄存器就是新上下文中的寄存器值。
sched 調用 swtch 切換到 cpu->scheduler，即 per-cpu 的調度器上下文。這個上下文是在之前 scheduler 調用 swtch時保存的。當 swtch 返回時，它不會返回到 sched 中，而是返回到 scheduler，其堆疊指標指向了當前 CPU 的調度器的堆疊，而非 initproc 的內核堆疊。

Code: Scheduling

進程想要讓出 CPU 必須要獲得進程表的鎖 ptable.lock，並釋放其擁有的其他鎖，修改自己的狀態（proc->state），然後調用 sched。yield和 sleep exit 都遵循了這個約定，我們稍後將會詳細研究。sched 檢查了兩次狀態，這裡的狀態表明由於進程此時持有鎖，所以 CPU 應該是在中斷關閉的情況下運行的。最後，sched 調用 swtch 把當前上下文保存在 proc->context 中然後切換到調度器上下文即 cpu->scheduler 中。swtch 返回到調度器堆疊中，就像是調度器調用的 swtch返回了一樣。調度器繼續其 for 迴圈，找到一個進程來運行，切換到該進程，然後繼續輪轉。
我們看到，在對 swtch 的調用的整個過程中，xv6 都持有鎖 ptable.lock：swtch 的調用者必須持有該鎖，並將鎖的控制權轉移給切換代碼。鎖的這種使用方式很少見，通常來說，持有鎖的執行緒應該負責釋放該鎖，這樣更容易讓我們理解其正確性。但對於上下文切換來說，我們必須使用這種方式，因為 ptable.lock 會保證進程的 state 和 context 在運行 swtch 時保持不變。如果在 swtch 中沒有持有 ptable.lock，可能引發這樣的問題：在 yield 將某個進程狀態設置為 RUNNABLE 之後，但又是在 swtch 讓它停止在其內核堆疊上運行之前，有另一個 CPU 要運行該進程。其結果將是兩個 CPU 都運行在同一個堆疊上，這顯然是不該發生的。
內核執行緒只可能在 sched 中讓出處理器，在 scheduler 中切換回對應的地方，當然這裡 scheduler 也是通過 sched 切換到進程中的。所以，如果要寫出 xv6 中切換執行緒的代碼行號，我們會發現一種執行規律，不斷迴圈。以這種形式在兩個執行緒之間切換的過程有時被稱作共行程式（coroutines）；在此例中，sched 和 scheduler 就是彼此的共行程式。
但在一種特殊情況下，調度器調用的切換到新進程的 swtch 不會在 sched 中結束。當一個新進程第一次被調度時，它從 forkret開始運行。之所以要運行 forkret，只是為了按照慣例釋放 ptable.lock；否則，這個新進程是可以就從 trapret 開始運行的。
scheduler運行了一個普通的迴圈：找到一個進程來運行，運行直到其停止，然後繼續迴圈。scheduler 大部分時間裡都持有 ptable.lock，但在每次外層迴圈中都要釋放該鎖（並顯式地允許中斷）。當 CPU 閒置（找不到 RUNNABLE 的進程）時這樣做十分有必要。如果一個閒置的調度器一直持有鎖，那麼其他 CPU 就不可能執行上下文切換或任何和進程相關的系統調用了，也就更不可能將某個進程標記為 RUNNABLE 然後讓閒置的調度器能夠跳出迴圈了。而之所以週期性地允許中斷，則是因為可能進程都在等待 I/O，從而找不到一個 RUNNABLE 的進程（例如 shell）；如果調度器一直不允許中斷，I/O 就永遠無法到達了。
scheduler 不斷迴圈尋找可運行，即 p->state == RUNNABLE 的進程。一旦它找到了這樣的進程，就將 per-cpu 的當前進程變數 proc 設為該進程，用 switchuvm 切換到該進程的頁表，標記該進程為 RUNNING，然後調用 swtch 切換到該進程中運行。
對於每個進程，調度維護了進程的一系列固定狀態，並且保證當狀態變化時必須持有鎖 ptable.lock。第一個固定狀態是，如果進程為 RUNNING 的，那麼必須確保使用時鐘中斷的 yield 時，能夠無誤地切換到其他進程；這就意味著 CPU 寄存器必須保存著進程的寄存器值（這些寄存器值並非在 context中），%cr3 必須指向進程的頁表，%esp 則要指向進程的內核堆疊，這樣 swtch 才能正確地向堆疊中壓入寄存器值，另外 proc 必須指向進程的 proc[] 槽中。另一個固定狀態是，如果進程是 RUNNABLE，必須保證調度器能夠無誤地調度執行該進程；這意味著 p->context必須保存著進程的內核執行緒變數，並且沒有任何 CPU 此時正在其內核堆疊上運行，沒有任何 CPU 的 %cr3 寄存器指向進程的分頁表，也沒有任何 CPU 的 proc 指向該進程。
正是由於要堅持以上兩個原則，所以 xv6 必須在一個執行緒中獲得 ptable.lock（通常是在 yield 中），然後在另一個執行緒中釋放這個鎖（在調度器執行緒或者其他內核執行緒中）。如果一段代碼想要將運行中進程的狀態修改為 RUNNABLE，那麼在恢復到固定狀態中之前持有鎖；最早的可以釋放鎖的時機是在 scheduler 停止使用該進程頁表並清空 proc時。類似地，如果 scheduler 想把一個可運行進程的狀態修改為 RUNNING，在該進程的內核執行緒完全運行起來（swtch 之後，例如在 yield 中）之前必須持有鎖。
除此之外，ptable.lock 也保護了一些其他的狀態：進程 ID 的分配，進程表槽的釋放，exit 和 wait 之間的互動，保證對進程的喚醒不會被丟失等等。

Code: Sleep and wakeup

總體思路是希望 sleep 將當前進程轉化為 SLEEPING 狀態並調用 sched 以釋放 CPU，而 wakeup 則尋找一個睡眠狀態的進程並把它標記為 RUNNABLE。
sleep 首先會進行幾個必要的檢查：必須存在當前進程並且 sleep 必須持有鎖（。接著 sleep 要求持有 ptable.lock。於是該進程就會同時持有鎖 ptable.lock 和 lk 了。調用者（例如 recv）是必須持有 lk 的，這樣可以保證其他進程（例如一個正在運行的 send）無法調用 wakeup(chan)。而如今 sleep 已經持有了 ptable.lock，那麼它現在就能安全地釋放 lk 了：這樣即使別的進程調用了 wakeup(chan)，wakeup 也不可能在沒有持有 ptable.lock 的情況下運行，所以 wakeup 必須等待 sleep 讓進程睡眠後才能運行。這樣一來，wakeup 就不會錯過 sleep了。
當 lk 就是 ptable.lock 的時候，這樣 sleep 在要求持有 ptable.lock 然後又把它作為 lk 釋放的時候會出現鎖死。
現在僅有該進程的 sleep 持有 ptable.lock，於是它通過記錄睡眠佇列，改變進程狀態，調用 sched讓進程進入睡眠。
稍後，進程會調用 wakeup(chan)。wakeup要求獲得 ptable.lock 並調用 wakeup1，其中，實際工作是由 wakeup1 完成的。對於 wakeup 來說持有 ptable.lock 也是很重要的，因為它也要修改進程的狀態並且要保證 sleep 和 wakeup 不會錯過彼此。而之所以要單獨實現一個 wakeup1，是因為有時調度器會在持有 ptable.lock 的情況下喚醒進程，。當 wakeup 找到了對應 chan 中處於 SLEEPING 的進程時，它將進程狀態修改為 RUNNABLE。於是下一次調度器在運行時，就可以調度該進程了。
wackup 必須在有一個監視喚醒條件的鎖的時候才能被調用。至於為什麼睡眠中的進程不會錯過喚醒，則是因為從 sleep 檢查進程狀態之前，到進程進入睡眠之後，sleep 都持有進程狀態的鎖或者 ptable.lock 或者是兩者兼有。由於 wakeup 必須在持有這兩個鎖的時候運行，所以它必須在 sleep 檢查狀態之前和一個進程已經完全進入睡眠後才能執行。
有些情況下可能有多個進程在同一佇列中睡眠；例如，有多個進程想要從管道中讀取資料時。那麼單獨一個 wakeup 的調用就能將它們全部喚醒。他們的其中一個會首先運行並要求獲得 sleep 被調用時所持的鎖，然後讀取管道中的任何資料。而對於其他進程來說，即使被喚醒了，它們也讀不到任何資料，所以喚醒它們其實是徒勞的，它們還得接著睡。正是由於這個原因，我們在一個檢查狀態的迴圈中不斷調用 sleep。
sleep 和 wakeup 的調用者可以使用任何方便使用的數位作為佇列號碼；而實際上，xv6 通常使用內核中和等待相關的資料結構的位址，譬如磁片緩衝區。即使兩組 sleep/wakeup 使用了相同的佇列號碼，也是無妨的：對於那些無用的喚醒，它們會通過不斷檢查狀態忽略。sleep/wakeup 的優點主要是其羽量級（不需另定義一個結構來作為睡眠佇列），並且提供了一層抽象（調用者不需要瞭解與之交互的是哪一個進程）。

Code: Pipes

實際上在 xv6 中有兩個使用 sleep/wakeup 來同步讀者寫者的佇列。一個在 IDE 驅動中：進程將未完成的磁片請求加入佇列，然後調用 sleep。中斷處理常式會使用 wakeup 告訴進程其磁片請求已經完成了。
更為複雜的例子是，我們從管道的一端寫入資料位元組，然後資料被拷貝到內核緩衝區中，接著就能從管道的另一端讀取資料了。
每個管道由一個結構體 struct pipe 表示，其中有一個鎖 lock和記憶體緩衝區。其中的域 nread 和 nwrite 表示從緩衝區讀出和寫入的位元組數。pipe 對緩衝區做了包裝，使得雖然計數器繼續增長，但實際上在 buf[PIPESIZE - 1] 之後寫入的位元組存放在 buf[0]。這樣就讓我們可以區分一個滿的緩衝區（nwrite == nread + PIPESIZE）和一個空的緩衝區（nwrite == nread），但這也意味著我們必須使用 buf[nread % PIPESIZE] 而不是 buf[nread] 來讀出/寫入資料。現在假設 piperead 和 pipewrite 分別在兩個 CPU 上連續執行。
pipewrite首先請求獲得管道的鎖，以保護計數器、資料以及相關不變數。piperead接著也請求獲得鎖，結果當然是無法獲得。於是它停在了 acquire上等待鎖的釋放。與此同時，pipewrite 在迴圈中依次寫入 addr[0], addr[1], …, addr[n-1] 並添加到管道中。在此迴圈中，緩衝區可能被寫滿，這時 pipewrite 會調用 wakeup通知睡眠中的讀者緩衝區中有資料可讀，然後使得在 &p->nwrite 佇列中睡眠的讀者從緩衝區中讀出數據。除此之外，sleep 在讓 pipewrite 的進程進入睡眠時還會釋放 p->lock。
現在 p->lock 被釋放了，piperead 嘗試獲得該鎖然後開始執行：此時它會檢查到 p->nread != p->nwrite（正是在 nwrite == nread + PIPESIZE 的時候 pipewrite 進入了睡眠），於是 piperead 跳出 for 迴圈，將資料從管道中拷貝出來，然後將 nread 增加讀取位元組數。現在緩衝區又多出了很多可寫的位元組，所以 piperead 調用 wakeup喚醒睡眠中的寫者，然後返回到調用者中。wakeup 會找到在 &p->nwrite 佇列上睡眠的進程，正是該進程之前在運行 pipewrite 時由於緩衝區滿而停止了。然後 wakeup 將該進程標記為 RUNNABLE。

Code: Wait, exit, and kill

sleep 和 wakeup 可以在很多需要等待一個可檢查狀態的情況中使用。父進程可以調用 wait 來等待一個子進程退出。在 xv6 中，當一個子進程要退出時它並不是直接死掉，而是將狀態轉變為 ZOMBIE，然後當父進程調用 wait 時才能發現子進程可以退出了。所以父進程要負責釋放退出的子進程相關的記憶體空間，並修改對應的 struct proc 以便重用。每個進程結構體都會在 p->parent 中保存指向其父進程的指標。如果父進程在子進程之前退出了，初始進程 init 會接收其子進程並等待它們退出。我們必須這樣做以保證可以為退出的進程做好子進程的清理工作。另外，所有的進程結構都是被 ptable.lock 保護的。
wait 首先要求獲得 ptable.lock，然後查看進程表中是否有子進程，如果找到了子進程，並且沒有一個子進程已經退出，那麼就調用 sleep 等待其中一個子進程退出（2439），然後不斷迴圈。除此之外，這裡 sleep 中釋放的鎖是 ptable.lock。
exit 首先要求獲得 ptable.lock 然後喚醒當前進程的父進程。這一步似乎為時過早，但由於 exit 這時還沒有把當前進程標記為 ZOMBIE，所以這樣並不會出錯：即使父進程已經是 RUNNABLE 的了，但在 exit 調用 sched 以釋放 ptable.lock 之前，wait 是無法運行其中的迴圈的。所以說只有在子進程被標記為 ZOMBIE之後， wait 才可能找到要退出的子進程。在退出並重新調度之前，exit 會把所有子進程交給 initproc。最後，exit 調用 sched 來讓出 CPU。
退出進程的父進程本來是通過調用 wait處於睡眠狀態中，不過現在它就可以被調度器調度了。對 sleep 的調用返回時仍持有 ptable.lock ；wait 接著會重新查看進程表並找到 state == ZOMBIE的已退出子進程。它會記錄該子進程的 pid 然後清理其 struct proc，釋放相關的記憶體空間。
子進程在 exit 時已經做好了大部分的清理工作，但父進程一定要為其釋放 p->kstack 和 p->pgdir；當子進程運行 exit 時，它正在利用 p->kstack 分配到的堆疊以及 p->pgdir對應的頁表。所以這兩者只能在子進程結束運行後，通過調用 sched 中的 swtch 被清理。這就是為什麼調度器要運行在自己單獨的堆疊上，而不能運行在調用 sched 的執行緒的堆疊上。
exit 讓一個應用程式可以自我終結；kill則讓一個應用程式可以終結其他進程。在實現 kill 時有兩個困難：1）被終結的進程可能正在另一個 CPU 上運行，所以它必須在被終結之前把 CPU 讓給調度器；2）被終結的進程可能正在 sleep 中，並持有內核資源。kill 很輕鬆地解決了這兩個難題：它在進程表中設置被終結的進程的 p->killed，如果這個進程在睡眠中則喚醒之。如果被終結的進程正在另一個處理器上運行，它總會通過系統調用或者中斷（例如時鐘中斷）進入內核。當它離開內核時，trap 會檢查它的 p->killed，如果被設置了，該進程就會調用 exit，終結自己。
如果被終結的進程在睡眠中，調用 wakeup 會讓被終結的進程開始運行並從 sleep 中返回。此舉有一些隱患，因為進程可能是在它等待的狀態尚為假的時候就從 sleep 中返回了。所以 xv6 謹慎地在調用 sleep 時使用了 while 迴圈，檢查 p->killed 是否被設置了，若是，則返回到調用者。調用者也必須再次檢查 p->killed 是否被設置，若是，返回到再上一級調用者，依此下去。最後進程的堆疊展開（unwind）到了 trap，trap 若檢查到 p->killed 被設置了，則調用 exit 終結自己。我們已經在管道的實現中看到了在 sleep 外的 while 迴圈中檢查 p->killed。
有一處的 while 沒有檢查 p->killed。ide 驅動直接重新調用了 sleep。之所以可以確保能被喚醒，是因為它在等待一個磁片中斷。如果它不是在等待磁片中斷的話，xv6 就搞不清楚它在做什麼了。如果有第二個進程在中斷之前調用了 iderw，ideintr 會喚醒該進程（第二個），而非原來等待中斷的那一個（第一個）進程。第二個進程會認為它收到了它正在等待的資料，但實際上它收到的是第一個進程想要讀的資料。
