Chapter 3 introduction

當一個 process 在運行時，CPU 會 execute 一個 normal processor loop：

  1. 讀取 instruction
  2. program counter 向前移動
  3. 執行 instruction

然而，user program 會使得某些 events transfer back to kernel 而非執行下一行 instruction，處理這些 event 會遇到以下三種問題：

  1. kernel 須讓 processor 從 user mode 轉移至 kernel mode
  2. kernel 和 device 須對應到 parallel activity
  3. kernel 須了解 device 的 interface

Systems calls, exceptions, and interrupts

遇到下列情形時，control 必須從 user program 轉移至kernel：

  1. 當 user program 想使用 operating system
  2. exception：當一個 program 做了 illegal action
  3. interrupt：當一個 device 為了引起 operating system 的注意，而產生一個訊號

在這三種情形中，設計 operating system 時，必須滿足以下要求：

  1. system 須可以存取 processor 的 register
  2. 在 kernel 中，system 是為了 execution 而建立
  3. system 要選一個位置讓 kernel 開始 execute
  4. kernel 要能夠擷取 enevt 的資料
  5. system 要維持 user process 和 kernel 之間的isolation

如果一個 operating system 有一個處理 interrupt 的 plan，那麼它也可以處理 system call 和 exception，其中，plan 的內容大致上如下：

  • interrupt 終止 normal processor loop，並且開始 execute 一個叫做 interrupt handler 的 sequence。
  • 在開始 execute interrupt handler 前，processor 會先儲存自己的 register ，如此一來，在 interrupt 結束後，才能回到原本的位置。processor 需要從 user mode 轉移到 kernel mode 再回到 user mode，是在轉入轉出 interrupt handler 時需要面對的問題。

在 x86 用的 interrupt，xv6 稱為 trap，此章節把這兩個詞交替使用，但是，它們在意義上，其實有些微的差異；interrupt：由 device 造成，可能跟 currently running process 無關；trap：由 currently running process 造成。

x86 protection

x86 有 4 個 protection levels，從 0 (most privilege) ~ 3 (least privilege)，大多數的 operation system 只會使用到 0 (kernel mode) 和 3 (user mode)，而 x86 executes instructions 的 current privilege level 會暫存於 register %cs。

在 x86，interrupt handler 被定義在 interrupt descriptor table(IDT)，IDT 有 256 個 entry，每個 entry 都有 %cs 和 %eip，用來處理相對應的 interrupt。

為了在 x86產生一個 system call，一個 program 呼叫 int n instruction，其中，n 代表 IDT 的 index，int instruction 實行如下：

  • 從 IDT 取得第 n 個 descriptor，其中，n 是 int 的 argumant
  • 確認 CPL <= DPL (Descriptor Privilege Level)
  • 只在 target segment selector 的 PL < CPL 時，於 CPU 內部 register 存入 %esp 和 %ss
  • 從某個 task segment descriptor 載入 %ss 和 %esp
  • push %ss
  • push %esp
  • push %eflags
  • push %cs
  • push %eip
  • 只在 interrupt 的情形下，清除 %eflags 裡的 IF (Instruction Fetch)
  • 設定 %cs 和 %eip 到 descriptor 中的 value

int instruction 是一個複雜的 instruction，它會思考以上所有的 action 是否是必要的。舉例來說，確認 CPL <= DPL 讓 kernel 防止 int 呼叫不適當的 IDT entry，在一個 user program 執行 int 時 ，IDT entry 的 DPL 一定為 3，如果 user program 沒有適當的 privilege，則 int 會導致 int 13 發生，這是一個很常見的 protection fault。

Code: The first system call

xv6 讓 x86 hardware 遇到一個會使 processor 產生 trap 的 int instrction 時， 做一些實際行動。x86 允許 256 個不同的 interrupt；0 ~ 31 是 software exception，而 xv6 map 32 個 hardware interrupt 到 32 ~ 63 這個 range 中，並且把 interrupt 64 設為 system call interrupt。









void tvinit(void)
{
    int i;
    for(i = 0; i < 256; i++)
        SETGATE(idt[i], 0, SEG_KCODE<<3, vectors[i], 0);
    SETGATE(idt[T_SYSCALL], 1, SEG_KCODE<<3, vectors[T_SYSCALL], DPL_USER);
    initlock(&tickslock, "time");
}
  • tvinit 是從 main 呼叫過來的 function，它建立一個 256 個 entry 的 table idt
  • interrupt i 存在 vector[i]中
  • tvinit 處理 TSYSCALL，user system 稱它為 trap，特別定義 gate 是一個 trap 的 type，trap gate 不會清除 IF flag，且它在 system 呼叫 handler 時，允許其它的 interrupt
  • kernel 也有設定一個 system call gate privilege 給 DPL_USER，DPL_USER 允許 user program 產生一個 int instruction 的 trap，然而，xv6 則否，若 xv6 常是如此，會有 general protection excrption 出現，即走向 vector 13

Code: C trap handler

































































void trap(struct trapframe *tf)
{
    if(tf−>trapno == T_SYSCALL)
    {
        if(myproc()−>killed)
            exit();
        myproc()−>tf = tf;
        syscall();
        if(myproc()−>killed)
            exit();
        return;
    }
    switch(tf−>trapno)
    {
        case T_IRQ0 + IRQ_TIMER:
         if(cpuid() == 0)
         {
            acquire(&tickslock);
            ticks++;
            wakeup(&ticks);
            release(&tickslock);
         }
         lapiceoi();
         break;
        case T_IRQ0 + IRQ_IDE:
         ideintr();
         lapiceoi();
         break;
        case T_IRQ0 + IRQ_IDE+1:
         break;
        case T_IRQ0 + IRQ_KBD:
         kbdintr();
         lapiceoi();
         break;
        case T_IRQ0 + IRQ_COM1:
         uartintr();
         lapiceoi();
         break;
        case T_IRQ0 + 7:
        case T_IRQ0 + IRQ_SPURIOUS:
         cprintf("cpu%d: spurious interrupt at %x:%x\n",
         cpuid(), tf−>cs, tf−>eip);
         lapiceoi();
         break;
        default:
         if(myproc() == 0 || (tf−>cs&3) == 0)
         {
            cprintf("unexpected trap %d from cpu %deip %x (cr2=0x%x)\n",
            tf−>trapno, cpuid(), tf−>eip, rcr2());
            panic("trap");
        }
        cprintf("pid %d %s: trap %d err %d on cpu %d ""eip 0x%x addr 0x%x−−kill proc\n",
        myproc()−>pid, myproc()−>name, tf−>trapno,
        tf−>err, cpuid(), tf−>eip, rcr2());
        myproc()−>killed = 1;
    }
    if(myproc() && myproc()−>killed && (tf−>cs&3) == DPL_USER)
        exit();
    if(myproc() && myproc()−>state == RUNNING &&
       tf−>trapno == T_IRQ0+IRQ_TIMER)
        yield();
    if(myproc() && myproc()−>killed && (tf−>cs&3) == DPL_USER)
        exit();
}
  • 每一個 handler 都有設定一個 trap frame，再呼叫 function trap
  • trap 用 hardware trap number tf->trapno 決定為什麼要被呼叫及需要做到什麼
  • 如果 trap 是 TSYSCALL，trap 呼叫 system call handler syscall
  • 在確認 system call 後，trap 會尋找 hardware interrupt
  • 如果 trap 不是一個 system call 也不是想引起注意的 hardware device，trap function 會假定此 interrupt 是由 incorrect behavior 造成
  • 如果造成 trap 的 code 是 user program，xv6 會 print 細節，並且將 proc->killed 設為 clean up user process

Code: System calls

















void syscall(void)
{
    int num;
    struct proc *curproc = myproc();
    num = curproc−>tf−>eax;
    if(num > 0 && num < NELEM(syscalls) && syscalls[num]) 
    {
        curproc−>tf−>eax = syscalls[num]();
    } 
    else 
    {
        cprintf("%d %s: unknown sys call %d\n",
        curproc−>pid, curproc−>name, num);
        curproc−>tf−>eax = −1;
    }
}
  • 在 system call 中，trap 會呼叫 syscall，syscall 從有存取 %eax 的 trap frame 載入 system call number，在對應到 system call table
  • 對於第一個 system call，%eax 有變數 SYS_exec，syscall 會呼叫 systam call 的 SYSexec entry，其中， SYS_exec 對應到被呼叫的 sys_exec
  • syscall 會用 %eax 記錄 system call function 的回傳值
  • 當 trap 回傳到 user space，它會載入 cp->tf 的值到 machine register，因此，當 exec 回傳時，它會回傳一個 system call handler 的回傳值
  • system call 一般會回傳一個負數來表示錯誤的情形，正數表示正確；如果 system call number 不合法，syscall 會印出 error 並且回傳 -1

Drivers

driver 是在 operating system 裡，用來處理特定 device 的code，它告訴 device hardware 如何 perform operation、配置 device 產生 interrupt、處理 resulting interrupt。
driver code 須巧妙的書寫，因為 driver 要會同時 execute，除此之外，driver 須了解 device 的 inerface。
disk driver 提供一個很好的例子，disk driver 從 disk 複製資料，再回到 disk，而 disk hardware 一般來說會將 disk 上的 data 看作 512 bytes 的 block 數列，其中，block 的 size 通常是 sector size的好幾倍，為了呈現xv6的block，有了struct buf。













struct buf 
{
    int flags;
    uint dev;
    uint blockno;
    struct sleeplock lock;
    uint refcnt;
    struct buf *prev;
    struct buf *next;
    struct buf *qnext;
    uchar data[BSIZE];
};
