OS HW1

運作原理

XV6可以在多執行緒的硬體上執行,當多個執行同時運作時,有機率會發生當A執行緒在讀取X檔案時B執行緒試著去改寫X檔案.當這種狀況發生時就可能會出現錯誤
xv6則透過lock來避免這種情形
lock限制同一時間只能有一個CPU持有這個lock,而當lock被一個CPU持有時,其他的CPU就不能持有這個lock
當code(CPU)獲得lock的持有權之後就可以確保在同一時間內只有持有lock的code會執行,以避免錯誤

代碼解釋

Race condition


















struct list {
    int data;
    struct list *next;
};

struct list *list = 0;

void
insert(int data)
{
    struct list *l;

    l = malloc(sizeof *l);
    l->data = data;
    l->next = list;
    list = l;
}

上述程式碼為一簡單的link list以及一個insert function
如果有兩個執行緒同時執行insert function
有可能會發生兩個執行緒同時執行到第15行(兩個執行緒都還沒有執行第16行)
這樣就會發生不確定l->next是指到那裏的狀況
這樣的情形就稱作Race condition




















struct list {
    int data;
    struct list *next;
};
struct list *list = 0;
struct lock listlock;

void
insert(int data)
{
    struct list *l;

    acquire(&listlock);
    l = malloc(sizeof *l);
    l->data = data;
    l->next = list;
    list = l;
    release(&listlock);
}

上述程式碼則是加入了lock
在acquire()到release()這之間的程式碼(13~18行)稱作critical section
當lock被一個執行緒持有時另一個就不能進入critical section
這樣就可以確保同時只能有一個CPU執行insert function,避免race condition

Code: Locks











void
acquire(struct spinlock *lk)
{
    for(;;) {
        if(!lk->locked) {
            lk->locked = 1;
            break;
        }
    }
}

上述程式碼為Lock中的spin lock的acquire function範例,當lk->locked為0時表示現在lk這個lock是可以被使用的
就會進到if中將狀態改為1(lk->locked=1)並跳出迴圈
如果lk->locked一直為1的話就會在迴圈中等待
但是如果只有上述的程式碼會發生一個狀況是兩個cpu同時執行到第5行,也會發生race condition
xv6的解決辦法是透過一個x86的指令 xchg ,透過這個指令在一個atomic operation會將register的data傳回memory, 就可以避免掉acquire自己本身的race condition

Code: Using locks
































iderw(struct buf *b)
{
  struct buf **pp;

  if(!holdingsleep(&b->lock))
    panic("iderw: buf not locked");
  if((b->flags & (B_VALID|B_DIRTY)) == B_VALID)
    panic("iderw: nothing to do");
  if(b->dev != 0 && !havedisk1)
    panic("iderw: ide disk 1 not present");

  acquire(&idelock);  //DOC:acquire-lock

  // Append b to idequeue.
  b->qnext = 0;
  for(pp=&idequeue; *pp; pp=&(*pp)->qnext)  //DOC:insert-queue
    ;
  *pp = b;

  // Start disk if necessary.
  if(idequeue == b)
    idestart(b);

  // Wait for request to finish.
  while((b->flags & (B_VALID|B_DIRTY)) != B_VALID){
    sleep(b, &idelock);
  }


  release(&idelock);
}

上述為iderw function,
在第5行會先檢查buffer b有沒有被lock住
然後再第12行會要求idelock以做後續的動作
第12行到第30行就是critical section
lock主要的目的就是要保持data的invarient
在每個執行續對data進行動作時都應保持著invarient
即便在過程中有暫時破壞掉data的invarient
在結束時都應回復data的狀態
lock就是在保護著critical section的invarient
避免在這個執行續持有lock,並還沒回覆狀態時另一個執行續就試著進入critical section

Deadlock and lock ordering
假設現在有兩個lock A跟B ,兩段程式碼X跟Y
如果X和Y都會使用到A和B,但是不同執行續分別在執行時
X先acquire A也要求到,Y先acquire B也成功acquire到
但在這時X和Y都無法要求到另一個lock
因為X和Y都還沒有把手上的lock release掉
這樣就稱作deadlock
為了避免這種狀況lock ordering就很重要
在acquire lock的順序在所有的程式碼中都需要一致
就可以避免掉另一段程式碼先把我要acquire的lock給佔去同時他要acquire我手上的lock

Interrupt handlers
在XV6中,當一個執行續持有一個lock時,會將該執行續的interrupts給disabled掉
為了被免當我持有一個lock是被interrupt,而interrupt的code去acquire我手上持有的這個lock
舉例:執行續執行iderw時持有著idelock,被interrupt而去執行ideintr,而ideintr會試著去acquire idelock,在這個狀況下,idelock永遠不會被release,因為只有iderw可以release他,而interrupt去執行的ideintr也acquire不到idelock而無法return,形成deadlock

Instruction and memory ordering
很多compiler為了效率會不依照程式碼的順序執行程式,而這樣的結果可能會導致錯誤







acquire(&listlock);
l = malloc(sizeof *l);
l->data = data;
l->next = list;
list = l;
release(&listlock);

以上述程式碼為範例,如果compiler先執行了第6行而讓其他執行續可以去執行第4行
這樣就會發生了錯誤
XV6為了避免這種情形,使用了__sync_synchronize()這個fuction來強制讓cpu照著程式碼的順序去執行

Sleep locks
有時XV6會需要持有一個lock很長一段時間,像是file system會在讀取以及寫入檔案時長時間持有lock
為了效率,XV6透過sleep-locks 可以在critical section時讓出processor
當執行續A持有lock L1時讓出processor,而執行續B希望acquire L1,我們需要保證A可以在B等待時執行完畢,release L1,在這時B不能透過spin-lock acquire L1, B需要將interrupts disabled的狀況下spin,以避免A無法執行,形成deadlock
XV6只有在file system有使用到sleep lock,在剩下的情況都是使用spin lock

struct sleeplock {
  uint locked;       //lock有沒有被持有
  struct spinlock lk; // 一個spinlock保護這個sleep lock
  
  // For debugging:
  char *name;        //lock的名字
  int pid;           // 持有lock的程序
};









struct spinlock {
  uint locked;       //lock有沒有被持有

  // For debugging:
  char *name;        //lock的名字
  struct cpu *cpu;   // 持有lock的cpu
  uint pcs[10];      // 鎖住lock的呼叫堆疊
                     











struct cpu {
  uchar apicid;                //APIC ID
  struct context *scheduler;   //swtch()進入scheduler
  struct taskstate ts;         //x86用來找interrupt的堆疊
  struct segdesc gdt[NSEGS];   // x86 global descriptor table
  volatile uint started;       // CPU是否開啟
  int ncli;                    // pushcli的深度
  int intena;                  // 在pushcki之前interrupt是否開啟
  struct proc *proc;           // 在這個cpu上執行的程序,沒有為null
};
















struct proc {
  uint sz;                     // 程序記憶體的size
  pde_t* pgdir;                // Page table
  char *kstack;                // 這個程序的最底層kernel stack
  enum procstate state;        // 程序的狀態
  int pid;                     // 程序ID
  struct proc *parent;         // 父程序
  struct trapframe *tf;        // Trap frame for current syscall
  struct context *context;     // swtch()以執行程序
  void *chan;                  // 非0, sleeping on chan
  int killed;                  // 非0,已被killed
  struct file *ofile[NOFILE];  // 打開的檔案
  struct inode *cwd;           // Current directory
  char name[16];               // Process name (debugging)
};
